# 题目名
Tree Shaking 原理与失效原因

## 题目描述
Tree Shaking 是如何工作的？为什么要求 ESM？常见“失效原因”有哪些？如何在工程中验证与治理？

## 题目答案
原理：
- Tree Shaking 基于静态分析：在构建时分析模块的导入/导出关系，标记未被引用的导出，并在生成代码时删除对应分支。

为什么要求 ESM：
- ESM 的 `import/export` 是静态结构（编译期可分析），而 CommonJS 的 `require` 是运行时行为，难以做可靠的静态裁剪。

常见失效原因：
- 模块存在副作用（side effects）：即使某个导出未被引用，模块顶层代码仍可能影响全局（打补丁、注册、改原型等），构建器不敢删除。
- 导出方式不利于静态分析：混用 CJS、动态导入路径、在运行时拼接导出等。
- Babel/TS 编译把 ESM 转成了 CJS（例如不当配置 `module`），导致后续无法 tree-shake。
- 使用了会触发全量引入的写法（例如某些库的 `import * as` / 默认导入聚合包）。

工程治理：
- 保持生产构建链路尽量输出 ESM 给打包器处理（不要过早转 CJS）。
- 明确声明副作用：在包的 `package.json` 配置 `sideEffects`（按真实情况标注哪些文件有副作用）。
- 用分析工具查看 bundle 内容（如构建产物分析），定位未裁剪的依赖与原因。
