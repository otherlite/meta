# 题目名
内容安全策略（CSP）的设计与落地

## 题目描述
什么是 CSP？如何用 CSP 降低 XSS 风险？`nonce`/`hash` 的使用方式是什么？上线时常见踩坑有哪些？

## 题目答案
CSP（Content-Security-Policy）通过响应头声明“允许加载/执行哪些资源”，从而把攻击面从“任意执行”收敛到“白名单内执行”。

核心能力：
- 限制脚本来源：`script-src 'self' https://trusted.cdn.com ...`
- 限制内联脚本：
  - 禁用 `unsafe-inline`
  - 用 `nonce-<random>` 或 `sha256-...` 允许特定内联脚本
- 限制 `eval`：默认建议禁用 `unsafe-eval`

`nonce` 与 `hash`：
- `nonce`：每次响应生成随机 nonce，内联脚本加 `nonce="..."` 才允许执行。
- `hash`：对内联脚本内容做哈希，写进 CSP 允许列表，内容变更需更新 hash。

落地踩坑：
- 三方 SDK 依赖内联脚本/动态执行（eval）导致报错，需要评估替代方案或单独收敛策略。
- 报警与灰度：先用 `Content-Security-Policy-Report-Only` 观察违规，再逐步收紧。
- 如果资源走多 CDN/多域名，需要完善白名单并避免过宽。
