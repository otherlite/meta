# 题目名
跨域的本质与 CORS 细节

## 题目描述
什么是同源策略？跨域限制具体限制了什么？CORS 的简单请求与预检请求分别是什么？`Access-Control-Allow-Credentials` 与 `withCredentials` 需要如何配合？

## 题目答案
同源策略：协议 + 域名 + 端口都相同才算同源。它主要限制浏览器在脚本层面对“跨源资源”的读取能力，防止恶意站点读取用户在其他站点的敏感数据。

跨域限制的核心：
- 不是“请求发不出去”，而是“响应是否允许被 JS 读取”。
- 例如 `<img src>`、`<script src>` 天然能跨域加载，但读取响应内容有严格限制。

CORS（Cross-Origin Resource Sharing）：服务器通过响应头声明哪些源被允许读取。

简单请求：
- 方法为 GET/HEAD/POST
- 且 `Content-Type` 只能是 `text/plain`、`multipart/form-data`、`application/x-www-form-urlencoded`
- 且请求头在“安全列表”内
- 结果：浏览器直接发请求，服务器回 `Access-Control-Allow-Origin` 等头。

预检请求（preflight）：
- 不满足简单请求条件时，浏览器先发一个 OPTIONS 请求询问权限。
- 服务端需要返回：
  - `Access-Control-Allow-Origin`
  - `Access-Control-Allow-Methods`
  - `Access-Control-Allow-Headers`
  - 可选 `Access-Control-Max-Age` 缓存预检结果

携带凭证（Cookie/HTTP 认证信息）：
- 前端：`fetch(url, { credentials: 'include' })` 或 XHR `withCredentials = true`
- 服务端：
  - 必须返回 `Access-Control-Allow-Credentials: true`
  - 且 `Access-Control-Allow-Origin` 不能是 `*`，必须是明确源（或动态回显但需校验）

其他跨域方案（了解即可）：反向代理、JSONP（仅 GET 且有安全限制）、postMessage。
