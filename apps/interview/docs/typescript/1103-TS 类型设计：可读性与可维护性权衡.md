# 题目名
TS 类型设计：可读性与可维护性权衡

## 题目描述
在大型项目中如何设计 TypeScript 类型，避免“类型体操”过度复杂？什么时候应该用更简单的类型，什么时候值得用更强的类型约束？

## 题目答案
目标不是“类型越强越好”，而是以合理成本换取更高的正确性与可维护性。

建议原则：
- 以业务边界为中心：对外部输入（接口响应、表单、URL 参数）要更严格；对内部局部变量可以适度放宽。
- 优先结构化与命名：把复杂条件类型拆成多个具名类型别名，提升可读性与复用性。
- 控制推导深度：嵌套的条件类型/递归类型会显著拖慢 TS 编译与 IDE 响应，必要时做降级。
- 明确“类型层的抽象层级”：
  - API 层：用 `unknown` + runtime 校验（zod/io-ts 等）再转为业务类型
  - 业务层：用 discriminated union 表达状态机
  - 组件层：用泛型表达可复用组件，但避免暴露过多内部细节

什么时候值得更强约束：
- 关键路径（支付、权限、路由、埋点协议）
- 需要高频重构的模块（类型可作为“编译期测试”）

什么时候应该降级：
- 类型复杂度明显超过业务复杂度
- 仅为“炫技”而引入的深层类型体操
- 导致编译时间、IDE 卡顿不可接受
