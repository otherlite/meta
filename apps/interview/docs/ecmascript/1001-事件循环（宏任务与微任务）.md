# 题目名
事件循环（宏任务与微任务）

## 题目描述
请说明浏览器中的事件循环（Event Loop）机制：宏任务/微任务的执行顺序是什么？常见的宏任务与微任务有哪些？给出一个包含 `Promise`、`setTimeout`、`queueMicrotask` 的例子并写出输出顺序。

## 题目答案
事件循环用于协调“同步代码、微任务队列、宏任务队列、渲染”等阶段。

- 执行顺序（简化版）：
  1. 执行一段同步 JS（一个 task 的同步部分）
  2. 同步执行结束后，清空微任务队列（microtask checkpoint），微任务可能继续产生新的微任务，直到队列清空
  3. 浏览器有机会进行渲染（不保证每次都渲染，但渲染发生在微任务清空之后、下一个宏任务之前）
  4. 取出下一个宏任务继续循环

- 常见宏任务（task）：`setTimeout`/`setInterval`、`postMessage`、`MessageChannel`（其回调是 task）、I/O（在部分环境）、UI 事件回调等。
- 常见微任务（microtask）：`Promise.then/catch/finally`、`queueMicrotask`、`MutationObserver`。

例子：

```js
console.log('A')

setTimeout(() => {
  console.log('B')
  Promise.resolve().then(() => console.log('C'))
}, 0)

Promise.resolve()
  .then(() => {
    console.log('D')
    queueMicrotask(() => console.log('E'))
  })
  .then(() => console.log('F'))

console.log('G')
```

输出顺序：
- 同步：`A`、`G`
- 微任务：`D`、`E`、`F`
- 下一轮宏任务：`B`
- 宏任务里的微任务：`C`

最终：`A G D E F B C`

面试加分点：
- “微任务清空后才可能渲染”的结论解释了为什么大量微任务会阻塞渲染（造成掉帧）。
- Node.js 的事件循环阶段与浏览器不同（但本题聚焦浏览器）。
