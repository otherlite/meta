# 题目名
垃圾回收机制与内存泄漏排查

## 题目描述
JavaScript 的垃圾回收（GC）大致怎么做？常见的回收算法/策略是什么？前端常见内存泄漏场景有哪些，如何排查？

## 题目答案
GC 的根本目标：找出“不可达对象”并释放内存。现代 JS 引擎以“可达性（reachability）”为核心。

基本概念：
- 根对象（roots）：全局对象、当前调用栈上的局部变量、闭包可达的变量等。
- 从根对象出发能遍历到的对象都是“可达”，不可达对象可回收。

常见策略（概念性回答即可）：
- 标记-清除（Mark-Sweep）：标记可达对象，再清除未标记对象。
- 分代回收：
  - 新生代（短命对象多）：更频繁回收，常用复制/Scavenge。
  - 老生代（长命对象多）：回收频率低，常用标记-清除/标记-整理（压缩碎片）。

前端常见内存泄漏来源：
- 事件监听/订阅未解除（组件卸载后仍被回调引用）。
- 定时器未清理（`setInterval`/长生命周期 `setTimeout`）。
- 闭包捕获大对象且长期存活（全局缓存、单例、队列）。
- Detached DOM：节点从 DOM 树移除但仍被 JS 引用。
- Map/数组无界增长：缓存不设上限（缺 LRU/TTL）。

排查手段：
- Chrome DevTools Memory：对比多次 Heap Snapshot，重点看 Retainers（谁在引用）。
- Performance 面板观察长时间交互后的内存趋势；关注节点数是否回落。
- 对可疑模块加“创建/销毁计数”与断点，定位未释放路径。
